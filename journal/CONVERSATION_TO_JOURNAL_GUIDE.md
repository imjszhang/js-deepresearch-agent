# 从 Cursor 对话生成 Journal 日记的规约

> 适用范围：把一次 Cursor Agent 对话中的问题、思考、方案、执行与验证，整理成符合 [`JOURNAL_TEMPLATE.md`](./JOURNAL_TEMPLATE.md) 的人类可读日记。  
> 目标读者：后续维护者、操作者、复盘者。  
> 输出位置：`journal/YYYY-MM-DD/<slug>.md`

---

## 1. 目标

Journal 不是聊天记录转写，也不是提交日志。它要回答四个问题：

| 问题 | 日记中对应内容 |
| --- | --- |
| 为什么做？ | 背景与动机 |
| 怎么想清楚的？ | 分析过程 |
| 最终怎么设计？ | 方案设计 |
| 实际做了什么，怎么确认？ | 实现要点、验证与测试 |

写作目标是让一个没有参与对话的人，在几分钟内理解：

- 当时遇到的真实问题是什么。
- 哪些方案被考虑过，为什么选当前方案。
- 关键实现落在什么文件和模块。
- 如何验证完成，以及还留下什么后续演化点。

---

## 2. 输入材料

生成 journal 前，应优先收集以下材料：

| 材料 | 用途 |
| --- | --- |
| 用户最初问题与后续追问 | 还原真实动机和需求变化 |
| 关键分析结论 | 写入「分析过程」，避免只写结果 |
| 最终计划或实施方案 | 写入「方案设计」和「关键决策」 |
| 实际修改文件 | 写入「实现要点」 |
| 测试命令与结果 | 写入「验证与测试」 |
| 遗留问题、下一步建议 | 写入「后续演化」 |

如果对话很长，不需要逐字重读。优先抓取：

- 用户明确提出的问题。
- 方案转折点。
- 被否定的备选方案。
- 最终落地的文件、命令、路径、产物。
- 测试是否通过。

---

## 3. 文件命名与位置

按日期创建目录：

```text
journal/
└── YYYY-MM-DD/
    └── <slug>.md
```

命名规则：

- `slug` 使用英文小写和连字符。
- 只描述主题，不写过长句子。
- 一天可以有多篇 journal。

示例：

```text
journal/2026-05-20/operator-intent-brief.md
journal/2026-05-20/execution-root-unification.md
```

---

## 4. 模板映射

必须以 [`JOURNAL_TEMPLATE.md`](./JOURNAL_TEMPLATE.md) 为结构基准。推荐映射如下：

| 模板章节 | 从对话中提炼什么 |
| --- | --- |
| 标题 | 用一句话概括本次工作，不超过 30 字为佳 |
| 元信息 | 日期、项目、类型、来源 |
| 背景与动机 | 用户为什么问、现象是什么、为什么当前机制不够 |
| 分析过程 | 搜索/阅读了哪些系统部位，发现了什么约束与根因 |
| 方案设计 | 最终方案、关键决策、备选方案对比 |
| 实现要点 | 文件结构、模块职责、关键数据流 |
| 验证与测试 | 运行了什么命令、结果是什么、是否有未验证风险 |
| 后续演化 | 近期可做的验证、扩展、清理或机制化改进 |

---

## 5. 写作流程

### 5.1 先归纳主线

先用四句话确定骨架：

```text
问题：这次要解决什么？
思考：为什么原方案不够，核心约束是什么？
方案：最终采用什么模型或实现？
执行：改了哪些地方，验证结果是什么？
```

这四句话可以放入文末「附：问题—思考—方案—执行对照」，也可作为写作提纲。

### 5.2 再填模板章节

不要按聊天时间线机械复述。应按读者理解顺序组织：

1. 先讲问题和动机。
2. 再讲关键分析。
3. 然后讲方案选择。
4. 最后讲实现、测试、后续。

### 5.3 保留决策理由

Journal 最有价值的部分不是「改了什么」，而是「为什么这样改」。

对每个关键选择，尽量写成表格：

```markdown
| 决策 | 选择 | 理由 |
| --- | --- | --- |
| 输入通道 | Operator Intent Brief | 一次性影响下一轮，不污染长期 guidance |
```

### 5.4 保留可操作信息

路径、命令、配置示例要具体：

```bash
jea intel brief list
jea intel brief processed --limit 10
```

文件路径用相对路径，优先指向仓库内文件：

```markdown
[`src/intelligence/operator-briefs.mjs`](../../src/intelligence/operator-briefs.mjs)
```

---

## 6. 内容边界

### 应该写

- 用户提出的真实问题。
- 系统已有机制为什么不够。
- 第一性原理或关键抽象。
- 关键文件与模块职责。
- 验证命令和结果。
- 后续可执行建议。

### 不应该写

- 大段逐字聊天内容。
- 无关的终端输出。
- 没有落地的随口想法。
- 未验证却写成事实的判断。
- 过度细碎的代码 diff。

---

## 7. 可读性要求

Journal 是给人读的，优先保证清晰。

### 7.1 基础要求

建议：

- 用现代白话书面语。
- 小节标题直白，不用隐喻。
- 长段落拆成列表或表格。
- 关键结论放在段首。
- 技术名词、文件名、命令用反引号。
- 对复杂流程可用 mermaid 图，但不要为了形式强行画图。

避免：

- 文言、玄学化标题、过度修辞。
- 「然后我又……」式流水账。
- 只列文件，不解释它们解决了什么问题。

### 7.2 推荐写作风格：微信公众号爆款文章

在不牺牲事实准确性的前提下，journal 推荐采用**微信公众号爆款技术文章**的写作风格：有问题感、有节奏、有结论、有转折，读起来像一篇给工程师看的复盘文章，而不是冷冰冰的变更清单。

核心风格：

| 维度 | 要求 |
| --- | --- |
| 开头 | 先抛出问题或冲突，让读者知道“为什么这事重要” |
| 叙事 | 用“现象 → 追问 → 发现 → 选择 → 落地”的推进方式 |
| 句子 | 短句优先，长句拆开；关键判断单独成段 |
| 标题 | 标题要有信息量和吸引力，但不能标题党 |
| 观点 | 每节开头先给结论，再展开证据 |
| 节奏 | 复杂处用表格、列表、代码块打断大段文字 |
| 人味 | 可以保留适度判断和语气，例如“这里真正的问题不是 X，而是 Y” |
| 边界 | 不夸大、不煽情、不把未验证猜测写成事实 |

推荐标题风格：

```markdown
# Operator Intent Brief：一次性人工意图，影响下一轮进化而不污染后续轮次
# 统一 Execution Root：修复 Agent 在错误目录里找文件
# 从“找不到文件”到资源寻址：一次 cwd 误判的系统性修复
```

推荐段落写法：

```markdown
真正的问题不是“模型有没有听到这句话”。

真正的问题是：这句话在进化系统里，到底属于事实、约束、意图，还是动作？

如果把一次性意图塞进长期 guidance，它会每轮反复生效；如果直接塞进 pending decisions，又绕过了 decide。两种方式都能“让系统做事”，但都会破坏系统自己的边界。
```

不推荐写法：

```markdown
本次修改新增了 operator-briefs.mjs，修改了 conversational-intel-pipeline.mjs，并新增 CLI 命令。
```

这种写法可以放在「实现要点」表格里，但不适合作为文章主线。主线应该解释**为什么需要这些修改**。

### 7.3 爆款风格与工程准确性的边界

允许：

- 用更强的问题意识组织文章。
- 用对比制造清晰转折。
- 用短句强调关键判断。
- 用“真正的问题是……”这类句式提炼核心矛盾。

不允许：

- 标题党。
- 夸大收益。
- 把猜测写成定论。
- 为了戏剧性省略关键证据。
- 把复杂工程问题简化成单一责任归因。

一句话原则：

> 写得要像爆款文章一样好读，但证据标准必须像工程复盘一样严格。

---

## 8. 质量检查清单

写完后逐项检查：

- [ ] 标题能一眼看出主题。
- [ ] 元信息完整：日期、项目、类型、来源。
- [ ] 背景里说明了用户为什么需要这项工作。
- [ ] 分析里保留了关键约束和被否定方案。
- [ ] 方案里有关键决策与理由。
- [ ] 实现要点包含主要文件和职责。
- [ ] 验证章节写明测试命令和结果。
- [ ] 后续演化是可执行建议，而不是空话。
- [ ] 没有把待验证假设写成事实。
- [ ] 路径和命令可被后续读者复用。

---

## 9. 推荐结尾

如果本次工作从对话中经历了明显的问题分析和方案落地，建议在文末增加一个简短对照表：

```markdown
## 附：本轮对话问题—思考—方案—执行对照

| 阶段 | 内容 |
| --- | --- |
| 问题 | ... |
| 思考 | ... |
| 方案 | ... |
| 执行 | ... |
```

这个表能帮助读者快速复盘整次对话的主线。
